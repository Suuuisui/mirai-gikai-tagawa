## 1. 最近の改善（2026-07-22）

- **議員・提出者から見るページ群を追加**: `/members`（議員ごとの賛否集計＋提出者区分別の入口）、
  `/members/[name]`（議員個人の投票記録）、`/proposers/[proposer]`（市長/議員/委員会提出の議案一覧）。
  `bills.member_votes`（121議案分）と議案本文の提出者行から導出しており、**DBスキーマ変更なし**。
  議案詳細には提出者バッジと議員名チップのリンクを追加
- **議案詳細のカバー画像を一覧カードと統一**: 一覧は動的カバー（BillCover）・詳細は旧静的画像という
  不一致を解消。クリック前後で同じ見た目になり、認知の一貫性が向上
- **議案詳細ヘッダーに会期チップ**: ステータス・提出日・提出者と並べて会期名を表示し、
  会期まとめページへリンク
- **モバイルヘッダーの「議会まとめ」をピル型ボタン化**（押せる見た目に）
- **AIチャットのサンプル質問を具体化**（「この議案で何が変わる？」等）
- **bill-repository の `.in()` を100件チャンク分割**: ID全件指定でURI長制限を超える問題の恒久対策

## 2. 課題（重要度順）

### 🔴 高: キーワード検索がない

611議案を探す手段がタグ・会期・提出者の一覧しかなく、「学校給食」「水道料金」のような
生活語彙から議案に到達できない。SEO流入以外の能動的な探索が実質不可能。

- 案: 議案名・summary を対象にしたクライアント側インクリメンタル検索（全件で数百KB程度なので
  APIなしでも成立する）。将来的には pg_trgm / full text search
- 置き場所: ヘッダーまたはトップの「本日は」帯の下

### 🟠 中: SNSシェア画像（OGP）がページの見た目と不一致

ページ内のカバーは動的カバーに統一したが、OGP画像は旧カテゴリ静止画像のまま。
シェアされたカードとページの第一印象が食い違う（今回のページ内不一致と同種の問題がSNS側に残っている）。

- 案: `@vercel/og`（ImageResponse）で動的カバーと同じデザインのOG画像を生成する
  `/api/og/[billId]` を追加し、metadata の images を差し替える

### 🟠 中: AIチャットが議員別賛否・新ページ群を知らない

「○○議員はどの議案に反対した？」に答えられない（コンテキストに member_votes が入っていない）。
新設した /members 系ページへの案内もできない。

- 案: 議案ページのチャットには当該議案の member_votes 要約を、トップのチャットには
  「議員・提出者から見るページの存在」をシステムプロンプトに追加

### 🟠 中: 議員ページが「姓のみ」で人物情報がない

出典画像の表記どおり姓のみのため、フルネーム・役職（議長等）・期などの人物情報がない。
同姓の別人が改選をまたぐと合算されるリスクも注記で逃げている状態。

- 案: 市議会公式の議員名簿から手動の対応表（姓→フルネーム・役職・在任期間）を
  `packages/seed/tagawa/` に作り、期を区切って集計する。顔写真は肖像権の扱いを確認するまで載せない

### 🟡 低: 提出者別一覧（市長提出517件）が1ページ全件表示

スクロールが長大で、ページ重量も大きい。年・会期での絞り込みやページネーションがない。

- 案: まず会期（年度）ごとの見出しでグルーピング、次点でページネーション

### 🟡 低: トップページから新ページ群への導線が弱い

/members への入口がハンバーガー・フッター・議案詳細の賛否セクションのみ。
トップに「議員・提出者から見る」セクション（例: 反対が多かった議員TOP等ではなく、中立に
会派カードを並べる）を置くと発見性が上がる。

### 🟡 低: 会期まとめページの賛否セクションから議員ページへ飛べない

「賛否が分かれた案件」セクション（split-vote-section）はバーのみで議員名を出していない。
議案詳細では議員名→個人ページのリンクを付けたので、会期まとめ側にも同様の導線があると回遊が繋がる。

### 🟡 低: 用語の市議会最適化が未了

ステータス表記が国会由来のまま（例: CompactBillCard の「成立」。市議会では「可決」が自然）。
「提出済み」「審議中」等も含め、市議会用語への総点検が未実施。

### 🟡 低: sitemap の member/proposer URL の lastModified が毎回 new Date()

クロール効率の観点では、データ更新時のみ変わる値（議案の max(updated_at) 等）が望ましい。

### 🟡 低: 既存の継続課題

- 「もっと詳しく」（難易度hard）が未整備のまま機能だけ残っている（前版から継続、方針未決定）
- 令和7年12月定例会 議案第63号が未収録（出典ページで確認できず）
- データ更新（新会期追加）が完全手動。定例会後に議決結果ページの更新を検知して
  通知する仕組みがあると運用漏れを防げる

## 3. 面白そうな機能アイデア（インパクト × 実装コストのメモ）

実装は未着手。政治的な解釈を誘導しないよう、出す場合は必ず出典と算出方法を明記すること。

| アイデア | 中身 | コスト | 備考 |
|---|---|---|---|
| 接戦議案ランキング | 賛成・反対の差が小さかった議案を全期間から一覧（9対8 など） | 小 | member_votes から計算可能。「もめた議案」への最短導線 |
| 議員の投票一致率 | 2議員間で同じ投票をした割合のマトリクス／「近い議員」表示 | 中 | 面白いが解釈注記が必須。会派の実態が見える |
| 会派別サマリー | 会派ごとの賛否傾向・所属議員・提出区分の内訳ページ | 小 | /members の集計を会派軸に並べ替えるだけ |
| 一般質問の収録 | 会議録から議員ごとの一般質問（テーマ・要旨）を抽出して個人ページに表示 | 大 | 「人」視点の本命。会議録スクレイパーの拡張が必要 |
| 議会カレンダー | 次の定例会日程・傍聴案内 | 小 | 市サイトの日程を手動転記でも成立 |
| 用語ミニ辞典 | 「専決処分」「委員会付託」等の平易な解説静的ページ | 小 | SEOに効く。AIチャットの回答からの内部リンク先にもなる |
