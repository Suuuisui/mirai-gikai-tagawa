## 2. fork運用の差分（AGENTS.mdより優先）

- **ベースブランチは `feature/tagawa-fork`**。upstreamの `develop` から分岐しないこと
- worktree作成: `git worktree add ../mirai-gikai-<name> -b <branch> feature/tagawa-fork`
- **PRは作らない**。検証（lint / typecheck / test）後、`git push fork <branch>:main` で直接本番リポジトリへ
- upstream（team-mirai）への push は**禁止**
- push すると自動で本番デプロイが走ることを忘れない

## 3. デプロイ

### 通常フロー

1. `git push fork <branch>:main`
2. GitHub Actions が動く（`gh run list -R Suuuisui/mirai-gikai-tagawa` で確認）
   - `Deploy to Vercel`: web / admin を並列で本番デプロイ
   - `Code Check`: lint / typecheck / build / test（**デプロイはこの結果を待たずに走る**ので、赤くなっていたら即修正すること）
   - `Integration Tests`: Supabase統合テスト
3. UI変更やデータ整合に関わる変更なら、**デプロイ完了後に** キャッシュ破棄（§5）

### VERCEL_TOKEN が失効したとき（頻出トラブル）

症状: Actions が `The token provided via --token argument is not valid` で失敗。

恒久対応（推奨・要人間作業）:

1. https://vercel.com/account/settings/tokens で「Create Token」（Scope: 自分のチーム、Expiration: No Expiration）
2. https://github.com/Suuuisui/mirai-gikai-tagawa/settings/secrets/actions の `VERCEL_TOKEN` を更新
3. 失敗したワークフローを「Re-run」（または `gh run rerun <run-id> -R Suuuisui/mirai-gikai-tagawa`）

暫定対応（CLIログインが生きている場合のみ）: `vercel login` 後、
`~/Library/Application Support/com.vercel.cli/auth.json` の `token` を `gh secret set VERCEL_TOKEN` に設定。
ただしCLI発行トークンは短命で、数日で再失効する。

## 4. データ更新手順（新しい会期を追加するとき）

データは Supabase が正で、`packages/seed/tagawa/` のスクリプト群から**全消し→再投入**方式で更新する。

### 手順

```bash
# 1. 田川市公式の「提出議案と議決結果」ページのURLと議案一覧を source-data.ts に転記
#    （packages/seed/tagawa/source-data.ts の SESSION_SOURCES に追記）

# 2. 会議録スクレイピング（1.5秒間隔・.cacheにキャッシュされる）と説明資料PDFリンク収集
pnpm --filter @mirai-gikai/seed tagawa:scrape          # → data/sessions.json 更新

# 3. 解説文を書く（bill-descriptions*.ts に追記。§6の執筆規約を参照）

# 4. CSV生成
pnpm --filter @mirai-gikai/seed tagawa:build-csv       # → packages/seed/csv/data/*.csv

# 5. （任意・保険）本番のAIインタビュー設定を手動バックアップ（§4.1参照）

# 6. 本番へ投入（.envに本番のSupabase接続情報を入れた状態で）
pnpm --filter @mirai-gikai/seed seed:csv               # 全テーブル削除→再投入
#    → インタビュー設定は自動でスナップショット→復元される（ログに「N件復元」と出る）

# 7. seed:csv のログで「インタビュー設定 N件復元 / 0件スキップ」を確認

# 8. コード変更もある場合は push → デプロイ完了を確認

# 9. ★必須★ キャッシュ破棄（§5）。必ずデプロイ完了「後」に実行
```

### 4.0 トップページ「注目の議案」の変更手順（手動キュレーション）

**本番は admin 画面での設定が正**。2026-07-24 以降は admin の
「トップページ」画面（`/homepage`）で、①注目の議案（候補一覧から追加・
ドラッグで並び替え）②タグ別セクション（表示タグと順番・各枠の自動選定
3件のプレビュー付き）をまとめて設定できる。保存時にwebのキャッシュも
自動で破棄される。

従来どおり議案の編集画面で「注目の議案」チェック＋「注目の表示順」
（小さいほど上、保存時に連番へ自動整列）を個別に設定してもよい。
両者は同じデータ（bills.is_featured / featured_priority）を編集している。

- `seed:csv` による再シード時も、admin設定は自動で保持される
  （インタビュー設定と同じスナップショット→議案名一致で復元。
  ログに「注目の議案の復元: N件復元 / M件スキップ」と出る）
- adminで1件でも注目設定がある場合、復元時にCSV初期値は全解除され
  admin設定だけが残る（admin側で外した議案が復活しない）
- 議案名を変更した場合は名前一致に失敗しスキップされる（警告ログが出るので
  admin画面で設定し直す）

`packages/seed/tagawa/featured-bills-data.ts` の `FEATURED_BILLS` 配列は
**初回投入・ローカル開発用のデフォルト**（DBに注目設定が1件も無いときだけ
有効になる）。配列の順番＝表示順。キーのタイポは build-csv 実行時に
「⚠️ FEATURED_BILLS のキーが…」と警告される。

### 4.0.1 トップページの表示のしくみ（何をどこで設定するか）

トップページには「注目の議案🔥」セクションと「タグ別（予算・条例…）」
セクションがあり、設定場所が異なる。

**注目の議案🔥**
- admin の議案編集で「注目の議案」にチェック＋「注目の表示順」を入力
- 表示順は保存時に自動で1からの連番に整列される（指定した位置に割り込み、
  未指定なら末尾に追加。他の議案の数値を手で直す必要はない）
- 再シードでも設定は自動保持（§4.0）

**タグ別セクション（各タグ3件表示）**
- **どのタグをどの順で出すか**: admin のタグ管理で `featured_priority` を設定
  したタグだけがトップに出る（昇順）。タグの説明文もここで編集できる。
  この2項目は再シードでも自動保持される（ラベル一致で復元。**タグ名自体を
  変えるとラベル一致に失敗する**ので、タグ名変更はseed側 build-csv の元データを
  直すこと）
- **各タグのどの議案3件が出るか**: これは手動設定ではなく**自動選定**。
  興味度スコア（`web/src/features/bills/shared/utils/interest-score.ts`。
  否決・不認定・賛否僅差・不信任等の話題性と新しさで加点）の高い順に3件。
  「注目の議案」に載せた議案は重複しないよう自動で除外される
- **セクションの並びの例外**: 直近90日以内に話題性の高い議案が出たタグは
  featured_priority より上に自動昇格する（sort-bills-by-tag-sections.ts）
- 特定の議案をタグ枠に必ず出したい場合は「注目の議案」に入れるのが確実
  （タグ枠は自動選定のため個別指定はできない）

**注意**: 議案へのタグ付け（bills_tags）と議案本文はseedデータ管理のため、
adminで変更しても再シードで元に戻る。恒久的に変えたい場合はseed側
（`packages/seed/tagawa/`）を修正すること。

### 4.1 インタビュー設定の自動保全と手動バックアップ（保険）

`packages/seed/shared/helper.ts` の `clearAllData()` は `interview_configs` / `interview_questions` を含む
**全テーブルを削除**するが、CSVにはこれらのデータが含まれない。このため 2026-07-16 以降の `seed:csv` は、
削除前にインタビュー設定を「会期slug::議案名」キーでスナップショットし、投入後に新しい bill_id へ
付け替えて**自動復元**する（`packages/seed/csv/interview-restore.ts`）。

また bill_id / diet_session_id は決定的UUID（uuid v5）で採番されるため、**再シードしてもURLは変わらない**
（2026-07-16以降。`packages/seed/tagawa/uuidv5.ts` の名前空間UUIDは変更厳禁）。

自動復元が「0件復元」になった場合（議案名の変更などでキー不一致）に備え、大きな更新の前は保険として
手動バックアップも取っておくとよい:

```bash
# 手動バックアップ（Supabase REST APIの例。KEYはservice_role JWT）
curl -s "$SUPABASE_URL/rest/v1/interview_configs?select=*" \
  -H "apikey: $KEY" -H "Authorization: Bearer $KEY" > interview_configs.backup.json
curl -s "$SUPABASE_URL/rest/v1/interview_questions?select=*" \
  -H "apikey: $KEY" -H "Authorization: Bearer $KEY" > interview_questions.backup.json

# 復元時: 対象議案の新IDを bills から name 一致で取得して bill_id を差し替え、
# POST（Prefer: resolution=merge-duplicates）で戻す
```

## 5. キャッシュの仕組みとrevalidate

web は `unstable_cache`（Vercel Data Cache）を多用しており、タグは `bills` / `diet-sessions` / `interview-configs`。

**落とし穴**: データ取得がエラーになった場合や0件を返した場合も、その「空の結果」がキャッシュされる。
そのため**データ投入・スキーマ変更・デプロイの後は必ずrevalidateを叩く**こと。

```bash
curl -X POST "https://mirai-gikai-tagawa-web.vercel.app/api/revalidate" \
  -H "Authorization: Bearer $REVALIDATE_SECRET" \
  -H "Content-Type: application/json" \
  -d '{"tags":["bills","diet-sessions","interview-configs"]}'
```

`REVALIDATE_SECRET` の値が手元にない場合は、Vercel CLI（`vercel login` 済みであること）で
本番の環境変数一式を取得できる。値をチャットやコミットに貼らないこと。

```bash
# リポジトリ直下（.vercel/project.json で mirai-gikai-tagawa-web にリンク済み）で実行
vercel env pull --environment=production /tmp/vercel-prod.env
```

**順序が重要**: コード変更を伴う場合、revalidate は**デプロイ完了後**に実行する。
デプロイ前に叩くと、旧コードのロジックでキャッシュが再構築されてしまい、修正が反映されない。

## 6. コンテンツ執筆規約（bill-descriptions*.ts）

議案の解説文は `packages/seed/tagawa/bill-descriptions.ts`（+ `-batch-b.ts` / `-batch-c.ts`）で管理する。

- キー形式: `"{会期key}:{議案番号ラベル}:{提出者}"`（例 `"r8-4-teirei:議案第28号:mayor"`）
- `source` の値: `"minutes"`（本会議録ベース）/ `"explanation-materials"`（説明資料・議決結果ページベース。会議録未公開時のフォールバック）
- `summary`: 1〜2文。「〜する議案。原案どおり可決されました。」の形
- `body`: 見出し付きMarkdownで構造化する。標準テンプレート:

```markdown
## ポイント解説
## なぜこの議案が出たのか
## 議会での主な論点          ← 会議録が無い議案では「## 具体的な内容」に置き換え
## この議案が可決（否決）されたことの意味
```

- body はバッククォートのテンプレートリテラル。**本文に `` ` `` や `${` を書かない**（書くならエスケープ）。
  末尾の `${NOTE_NO_MINUTES}` は「会議録未公開」の注記変数なので消さない
- 事実はソース（会議録・説明資料PDF）に忠実に。**ソースにない金額・日付・固有名詞を書かない**
- 会議録が公開されたら `explanation-materials` の議案を `minutes` ベースで書き直す（論点見出しを追加できる）

## 7. AIインタビュー運用

- 管理画面（admin）で議案ごとに `interview_configs`（mode: loop/bulk/targeted、status: public/closed）と
  質問7問を設定。Q1/Q2のクイックリプライはLLM生成、Q3-Q7は固定文
- 既定AIモデルのコストは1インタビューあたり約29円。AI GatewayキーはVercelダッシュボードで月額上限を設定済み
- Supabase Auth の匿名ログイン（`external_anonymous_users_enabled`）が有効であることが前提
- **再シードで消える**ことに注意（§4.1）

## 8. トラブルシューティング

| 症状 | 原因と対処 |
|---|---|
| トップページ・一覧が空になる | ①キャッシュに空結果が焼き付いた → §5のrevalidate。②過去に難易度トグルを触ったブラウザのCookie（`bill_difficulty_level=hard`）→ 2026-07-16のコミットでサーバー側をnormal固定にして解決済み |
| Actionsデプロイ失敗 `token is not valid` | §3のVERCEL_TOKEN更新 |
| Supabase接続で Invalid API key | 新形式 `sb_secret_` キーの既知不具合 → 旧 service_role JWT を使う |
| ローカルで `pnpm test` がrubyful関連で落ちる | ローカル環境起因（localStorage周り）。CIの Code Check では同じテストが通っており、本番影響なし |
| 議案詳細のURLが404になった | 2026-07-16以降は決定的UUIDのため再シードでURLは変わらない（§4.1）。404になるのは議案名・会期キー・提出者を変更した場合のみ（IDの元ネタが変わるため） |
| 管理画面にログインできない | Supabaseダッシュボード（Auth > Users）からパスワードリセット。管理者は `app_metadata.roles=["admin"]` が必要 |
