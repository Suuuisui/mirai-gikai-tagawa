# 議員提出資料PDF活用の仕組み設計（未実装・設計メモ）

**ステータス: 未実装。トークン予算の都合で設計のみ記録し、実装は後日着手する。**

## 背景・目的

現状の議案解説文（`bill-descriptions*.ts`）は、本会議録（会議録検索システム）を主な一次資料としている。しかし田川市公式サイトの「〜の提出議案と議決結果」ページ（例: https://www.joho.tagawa.fukuoka.jp/kiji0037791/index.html ）には、議案ごとに市長・委員会・議員が実際に議会へ提出した「説明資料」PDFが添付されており、これを取り込めば以下が改善できる：

- 会議録が本稿執筆時点で未公開の議案（令和8年5月以降の一部会期）でも、公式の一次資料に基づいた解説が書ける（現状は `source: "explanation-materials"` として簡易な記述に留めている）
- 会議録には載らない、提出時点の趣旨説明・積算根拠・図表等の詳細を解説文に反映できる
- 議案詳細ページに「一次資料への直接リンク」を張れる（過去の比較検討で「議案ごとの出典リンクが無い」と指摘した箇所の解消にもなる）

## 対象ページの構造（実地調査済み）

### ページ単位
`session.sourceUrl`（`packages/seed/tagawa/source-data.ts`の`SessionSource.sourceUrl`）が指す「〜の提出議案と議決結果」ページそのものに、その会期の全議案の説明資料PDFへのリンクがまとまっている。**新たなURL探索は不要で、既に保持しているセッションごとのURLをそのまま使える。**

### PDFリンクの形式
```html
<a href="https://www.joho.tagawa.fukuoka.jp/kiji0037791/3_7791_24116_up_4lrp6cvq.pdf">
  &nbsp;59号　説明資料&nbsp;（261キロバイト）
</a>
```

実例（令和3年12月定例会 `kiji0037791` より抜粋）:

| リンクテキスト | PDF URL |
|---|---|
| `59号　説明資料` | `.../3_7791_24116_up_4lrp6cvq.pdf` |
| `議員提出議案第16号` | `.../3_7791_24118_up_bqedw8bp.pdf` |
| `60号　厚生委員会` | `.../3_7791_24201_up_uq3lpmyg.pdf` |
| `60号　建設経済委員会` | `.../3_7791_24202_up_jj6b4ggk.pdf` |
| `63号、65-71号　説明資料` | `.../3_7791_24205_up_8s7lgy4w.pdf`（**複数号を1PDFにまとめているケースあり**） |

URLパターン: `https://www.joho.tagawa.fukuoka.jp/{sourceUrlのkiji部分}/3_{数字}_{連番}_up_{8文字英数ハッシュ}.pdf`

### パース方針
リンクテキストから号数を正規表現で抽出する。パターンは大きく3種類:

1. `^(\d+)号` → 市長提出議案の号数（`議案第{N}号`に変換して`billNumberLabel`と突合）
2. `議員提出議案第(\d+)号` → そのまま`billNumberLabel`と突合
3. `(\d+)号、(\d+)-(\d+)号` 等の複数号まとめ表記 → 範囲展開して複数の`billNumberLabel`に同一PDFを紐付け（1つの資料が複数議案を横断的に説明しているケース。分割は困難なため「複数議案共通の資料」として扱う）

同一号数に対し「説明資料」と「厚生委員会」「建設経済委員会」のように委員会ごとに複数PDFがあるケースもある（配列で保持する）。

## 実装ステップ案（フェーズ分け）

### フェーズ1（低コスト・即効性あり）: URL収集とリンク表示のみ
1. `scrape.ts`に、既存のページ取得処理（`fetchWithCache`相当。1.5秒間隔・キャッシュ機構は流用）を再利用してPDFリンクを抽出する関数を追加。
2. `BillSource`型に `explanationMaterialUrls?: { label: string; url: string }[]` を追加し、`data/sessions.json`に保存。
3. `build-csv.ts`で議案詳細ページ用データに引き渡し、`web`側の議案詳細ページ（`BillDisclaimer`付近、または新規セクション）に「提出時の説明資料（PDF）」リンクとして表示する。
   - この段階ではAI生成は一切不要。公式サイトへの導線を増やすだけなので事実誤認リスクがない。
   - コード変更量: 小〜中（スクレイパー拡張＋型追加＋表示コンポーネント1つ）。

### フェーズ2（コスト大）: PDF本文をAI解説文生成の情報源に組み込む
1. PDFをダウンロードし、テキスト抽出（`pdf-parse`等のnode向けライブラリを検討。スキャンPDF＝画像ベースの場合はテキスト抽出できないため、その場合はフェーズ1のリンク表示のみに留める）。
2. 会議録ベースの解説文生成フロー（既存の`bill-descriptions.ts`執筆パターン、Sonnetエージェントへの委譲）と同様に、PDF抽出テキストを情報源として渡し、`source: "explanation-materials"`（既存の値をそのまま使う）として解説文を書く。
   - 優先対象は「会議録が未公開で`explanation-materials`が唯一の情報源になっている議案」（令和8年5月臨時会以降）。ここが最も解像度が上がる箇所。
   - 既に`source: "minutes"`で会議録ベースの解説がある議案は、PDF資料を補助情報として使うかどうかは費用対効果次第（優先度低）。

## 注意点・リスク

- **サイトへの負荷配慮**: 既存の`scrape.ts`の1.5秒間隔ルールをPDF取得にもそのまま適用する。PDFは数百KB〜数MBあるため、テキスト抽出前提でダウンロードする場合は取得件数に応じて時間がかかる（611議案の一部が対象になるとしても、セッション単位でまとめて取得すれば重複は避けられる）。
- **スキャンPDFの可能性**: 古い年度の資料は画像スキャンPDFの可能性があり、その場合テキスト抽出できない。OCRまで実施するかは別途判断。
- **複数号まとめPDFの帰属**: 上記の「63号、65-71号」のようなケースは、どの号にどこまで言及しているか機械的に切り分けられないため、AIに渡す際は「この資料は複数議案に関する共通資料」である旨を明示して誤帰属を防ぐ。
- **PDFの著作権・転載範囲**: 公式資料へのリンクは問題ないが、PDF全文を自サイトに転載・保存することはせず、あくまで解説文生成の情報源として使い、リンクは公式サイトへ張る（現状の会議録の扱いと同じ方針）。
- **リポジトリへの生PDFコミットは行わない**: ダウンロードしたPDFはビルド時の一時キャッシュ（`.cache`ディレクトリ、`scrape.ts`が既に使っている仕組み）に留め、コミット対象にしない。

## 見積もり感

- フェーズ1: 既存の`scrape.ts`基盤に乗るため、実装自体は半日程度の規模。
- フェーズ2: PDF本文抽出＋AI解説文生成を令和8年5月以降の対象議案（数件〜十数件程度）に絞れば、これまでの構造化バッチ作業と同程度の規模感。全611件を対象にするのは非現実的。
